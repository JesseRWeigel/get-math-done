# Category Theory Protocols

> Step-by-step methodology guides for categorical reasoning and proof construction.

## Protocol: Functor Verification

### When to Use
Defining or verifying functors between categories, including covariant, contravariant, and bifunctors.

### Steps
1. **Specify the source and target categories** — name the categories, their objects, and morphisms explicitly
2. **Define the object map** — for each object A in the source, specify F(A) in the target
3. **Define the morphism map** — for each morphism f: A → B, specify F(f): F(A) → F(B) (covariant) or F(f): F(B) → F(A) (contravariant)
4. **Verify identity preservation** — check F(id_A) = id_{F(A)} for all objects A
5. **Verify composition preservation** — check F(g ∘ f) = F(g) ∘ F(f) (covariant) or F(g ∘ f) = F(f) ∘ F(g) (contravariant)
6. **For bifunctors**: verify functoriality in each variable separately, then check the interchange condition
7. **Provide a concrete example** — instantiate with at least one non-trivial morphism

### Common LLM Pitfalls
- Reversing the direction of morphism maps for contravariant functors
- Forgetting to verify the identity axiom (assuming it is "obvious")
- Defining the functor on objects only and neglecting the morphism map
- Confusing covariant and contravariant Hom-functors: Hom(A, −) is covariant, Hom(−, B) is contravariant

## Protocol: Natural Transformation Construction

### When to Use
Defining, verifying, or reasoning about natural transformations between functors.

### Steps
1. **Identify the two functors** F, G: C → D between the same pair of categories
2. **Define the components** — for each object A in C, specify η_A: F(A) → G(A) in D
3. **Verify the naturality square** — for every morphism f: A → B in C, check that G(f) ∘ η_A = η_B ∘ F(f)
4. **Draw the naturality square diagram** — make it explicit with all four morphisms labeled
5. **Check naturality for generating morphisms** — if the category is generated by a known set of morphisms, checking naturality on generators suffices
6. **For natural isomorphisms**: verify each component η_A is an isomorphism and construct the inverse transformation

### Common LLM Pitfalls
- Checking the naturality condition for only one morphism and claiming it holds generally
- Confusing the direction of the naturality square (getting the composition order wrong)
- Forgetting that natural transformations between functors to Set are determined by the Yoneda lemma
- Defining components that are not actually morphisms in the target category

## Protocol: Adjunction Verification

### When to Use
Establishing that a pair of functors forms an adjunction (F ⊣ G), verifying unit/counit, or using adjunctions to transport results.

### Steps
1. **State the adjunction** — identify the left adjoint F: C → D and right adjoint G: D → C
2. **Choose the verification method**:
   - **Hom-set bijection**: show Hom_D(F(A), B) ≅ Hom_C(A, G(B)) naturally in A and B
   - **Unit/counit**: construct η: Id_C → GF (unit) and ε: FG → Id_D (counit) satisfying triangle identities
3. **For Hom-set approach**: verify naturality in both variables, not just bijectivity for a single pair
4. **For unit/counit approach**: verify both triangle identities:
   - εF ∘ Fη = id_F (left triangle)
   - Gε ∘ ηG = id_G (right triangle)
5. **Derive consequences** — left adjoints preserve colimits, right adjoints preserve limits
6. **Check for uniqueness** — adjoints are unique up to natural isomorphism

### Common LLM Pitfalls
- Verifying only one triangle identity and assuming the other follows
- Confusing which functor is left vs right adjoint (left = "free" construction, right = "forgetful" typically)
- Forgetting naturality when establishing the Hom-set bijection
- Misapplying the RAPL theorem (right adjoints preserve limits) to a non-adjoint situation

## Protocol: Limits and Colimits

### When to Use
Computing or proving existence of limits (products, equalizers, pullbacks, terminal objects) or colimits (coproducts, coequalizers, pushouts, initial objects).

### Steps
1. **Identify the diagram shape** — discrete (products/coproducts), parallel pair (equalizers/coequalizers), span (pullbacks), cospan (pushouts)
2. **State the universal property** precisely — the limit/colimit is an object L with a cone/cocone that factors uniquely through any other cone/cocone
3. **Construct the candidate** — build the object and projection/injection morphisms explicitly
4. **Verify the universal property** — given any other cone/cocone, construct the unique mediating morphism
5. **Prove uniqueness** of the mediating morphism — typically by showing two mediating morphisms must agree on generators or by diagram chase
6. **For existence in specific categories**: use the fact that Set, Grp, Top, etc. have all small limits and colimits; construct them concretely
7. **Check preservation** — do relevant functors (forgetful, Hom, tensor) preserve this limit/colimit?

### Common LLM Pitfalls
- Constructing a cone but failing to verify the universal property (existence without uniqueness)
- Confusing limits with colimits — limits use projections, colimits use injections
- Forgetting that filtered colimits commute with finite limits in Set (but not in general categories)
- Assuming a functor preserves limits without checking (only right adjoints automatically preserve limits)

## Protocol: Yoneda Lemma Application

### When to Use
Using the Yoneda lemma to compute natural transformations, embed categories, or prove representability.

### Steps
1. **State the version needed** — covariant Yoneda: Nat(Hom(A, −), F) ≅ F(A), or contravariant (presheaf) version
2. **Identify the functor F** and the representing object A
3. **Apply the bijection** — a natural transformation α: Hom(A, −) → F is determined by α_A(id_A) ∈ F(A)
4. **For the Yoneda embedding** y: C → [C^op, Set]: verify it is full and faithful (objects are determined by their morphisms)
5. **For representability**: to show F is representable, find an object A and a natural isomorphism Hom(A, −) ≅ F
6. **Use Yoneda to prove equalities** — two morphisms f, g: A → B are equal iff Hom(B, −) cannot distinguish them (i.e., h ∘ f = h ∘ g for all h)

### Common LLM Pitfalls
- Applying the covariant Yoneda lemma when the contravariant version is needed (or vice versa)
- Forgetting that the bijection in Yoneda is natural in both A and F
- Misidentifying the representing object when checking representability
- Confusing the Yoneda embedding with other fully faithful functors
